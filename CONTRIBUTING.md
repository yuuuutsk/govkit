# コントリビューションガイド

govkit へのコントリビューションに興味を持っていただき、ありがとうございます！

## 行動規範

このプロジェクトに参加するすべての人は、相互尊重と建設的な議論を心がけてください。

## コントリビューションの方法

### バグ報告

バグを見つけた場合は、以下の情報を含めて Issue を作成してください：

- バグの詳細な説明
- 再現手順
- 期待される動作
- 実際の動作
- 使用している OS と Go のバージョン
- 可能であれば、エラーメッセージやスクリーンショット

### 機能リクエスト

新機能の提案は歓迎します！以下の情報を含めて Issue を作成してください：

- 機能の詳細な説明
- なぜその機能が必要なのか
- 可能であれば、使用例やモックアップ

### プルリクエスト

プルリクエストを作成する前に、以下を確認してください：

1. **Issue を作成する**: 大きな変更の場合は、まず Issue で議論してください
2. **フォークする**: このリポジトリをフォークしてください
3. **ブランチを作成する**: `git checkout -b feature/your-feature-name`
4. **コードを書く**: コーディング規約に従ってください
5. **テストを追加する**: 新機能にはテストを追加してください
6. **テストを実行する**: `go test -v ./...` ですべてのテストが通ることを確認
7. **コミットする**: わかりやすいコミットメッセージを書いてください
8. **プッシュする**: `git push origin feature/your-feature-name`
9. **PR を作成する**: GitHub でプルリクエストを作成してください

## コーディング規約

### Go のコーディングスタイル

- `gofmt` でフォーマットしてください
- `golint` で警告が出ないようにしてください
- 関数やメソッドには適切なコメントを付けてください
- エラーハンドリングを適切に行ってください

### コミットメッセージ

コミットメッセージは以下の形式で書いてください：

```
<type>: <subject>

<body>
```

**Type:**
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響しない変更（空白、フォーマットなど）
- `refactor`: バグ修正でも機能追加でもないコード変更
- `test`: テストの追加や修正
- `chore`: ビルドプロセスやツールの変更

**例:**
```
feat: CSV の行列反転機能を追加

ユーザーがボタンをクリックすることで、CSV テーブルの
行と列を簡単に入れ替えられるようになりました。
```

## テスト

新しいコードには必ずテストを追加してください。

```bash
# すべてのテストを実行
go test -v ./...

# カバレッジを確認
go test -cover ./...
```

## ビルド

```bash
# ビルド
make build

# クリーンアップ
make clean
```

## ドキュメント

コードの変更に伴い、README.md や関連ドキュメントも更新してください。

## レビュープロセス

1. プルリクエストが作成されると、CI が自動的に実行されます
2. すべてのテストが通る必要があります
3. コードレビューが行われます
4. レビューで承認されたら、メインブランチにマージされます

## 質問がある場合

質問がある場合は、遠慮なく Issue を作成してください。
